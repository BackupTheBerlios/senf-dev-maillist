Jens <jens.moedeker at fokus.fraunhofer.de> writes:
> A few comments to your point from my side.
> The Makefile is rather for people which are not familiar with scons, so
> almost everybody :)

Hmm.. I don't like this much. Not in the repository. I don't care what
anyone is doing locally, but writing a makefile just so typing 'make'
will do the correct thing ...

> For the distribution version I think SENF should not demand to use
> scons or any other utility. The package should just provide library,
> header, and documentation files possibly examples.

I have just debianized the library completely. Am just checking, that
everything is working to commit into the repository. I build 5 debian
packages:
   
   libsenf-doc
   libsenf-ppi-dev
   libsenf-packets-dev
   libsenf-scheduler-dev
   libsenf-socket-dev
   libsenf-utils-dev

with correct dependencies. They install the includes to /usr/include,
the libraries to /usr/lib. 

> Right now the files are split into the library parts (socket, packet,
> schedule, utils, ppi). I think it good to keep this separate for
> documentation purpose, but for the include files I don't feel that
> comfortable with bothering the user with include paths like
> 'senf/Packet/Packet.hh'. Unfortunately I expect that this
> restructuring will cost a lot of efforts.

I have big problems with throwing everything into one directory. Not
(only) because it would be a lot of work, I don't think it is
sensible. What may be sensible is to have 'top-level-includes'
directly unter /usr/include or in /usr/include/senf named after the
library name which include the basic stuff needed for the library.

stefan.
